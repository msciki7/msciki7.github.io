# 데이터(Data)와 정보(Information)

&ensp;데이터(Data)<br/>
* 현실 세계에서 관찰되거나 측정되어 수집된 사실이나 값
* 가공되지 않은 상태

&ensp;예<br/>
```
25
홍길동
서울
```

&ensp;정보(Information)<br/>
* 데이터를 처리하여 의미를 부여한 결과

&ensp;예<br/>
```
홍길동의 나이는 25세이다
서울 거주자 평균 나이는 27세이다
```

<p align="center"><img src="/assets/img/database/1주차/1-1.png" width="600"></p>

&ensp;핵심 차이<br/>

| 구분          | 의미                |
| ----------- | ----------------- |
| Data        | 가공되지 않은 사실        |
| Information | 데이터를 처리한 의미 있는 결과 |

# 데이터베이스(Database)

&ensp;정의: 특정 조직의 여러 사용자가 공유하여 사용할 수 있도록 통합해서 저장한 운영 데이터의 집합<br/>

<p align="center"><img src="/assets/img/database/1주차/1-2.png" width="600"></p>

&ensp;데이터베이스의 특징<br/>
1. 통합 데이터(Integrated Data)
* 데이터 중복 최소화
2. 저장 데이터(Stored Data)
* 컴퓨터 저장 매체에 저장된 데이터
3. 공유 데이터(Shared Data)
* 여러 사용자가 함께 사용
4. 운영 데이터(Operational Data)
* 조직 운영을 위해 사용되는 데이터

# 데이터베이스의 필요성

&ensp;파일 시스템의 문제점<br/>
&ensp;1. 데이터 중복<br/>
&ensp;같은 데이터가 여러 파일에 존재<br/>
```
파일A → 고객정보
파일B → 고객정보
```

&ensp;문제: 데이터 불일치 발생<br/>

&ensp;2. 데이터 종속성<br/>
&ensp;데이터 구조 변경 시<br/>
```
데이터 파일 변경 → 프로그램도 수정
```

&ensp;3. 보안 문제<br/>
* 데이터가 여러 파일에 분산
* 접근 통제 어려움

# 데이터베이스 시스템(DB System)

&ensp;구성요소<br/>
&ensp;데이터베이스 시스템은 3가지로 구성<br/>
<p align="center"><img src="/assets/img/database/1주차/1-3.png" width="600"></p>

1. 데이터베이스(Database)
* 실제 데이터 저장
2. DBMS(Database Management System)
* 데이터 관리 소프트웨어
3. 데이터 모델(Data Model)
* 데이터를 표현하는 구조

## 데이터베이스 시스템의 특징

&ensp;동시 공유(Concurrency)<br/>
&ensp;여러 사용자와 프로그램이 **동시에 데이터 접근 가능**<br/>
&ensp;예<br/>
```
사용자 A → 조회
사용자 B → 수정
사용자 C → 검색
```

* 조직에서 사용하던 데이터를 통합하여 저장한 공유 데이터
* 실시간 서비스로 실시간 접근성 보장
    - 사용자가 데이터 요청 시 수 초 내에 제공
* 저장된 데이터 값은 지속적인 변화 발생
* 파일 시스템 문제점 극복
    - 데이터 통합에 따른 데이터 중복 최소화
* 동시 공유 보장
* 저장된 데이터 값에 따라 참조
    - 사용자에 따라 접근할 수 있는 데이터의 속성을 다르게 관리 가능

# 파일 시스템 vs DBMS

<p align="center"><img src="/assets/img/database/1주차/1-4.png" width="600"></p>

&ensp;DBMS의 장점<br/>
<p align="center"><img src="/assets/img/database/1주차/1-5.png" width="600"></p>

* 데이터 복구, 보안(암호화), 동시성 제어, 데이터 관리 기능 등 제공
* 데이터 무결성 유지 및 데이터 표준 준수 용이
    - 무결성 유지를 위한 다양한 제약 조건
    - 보안기술(해쉬, 맥 등) 활용 데이터 저장
    - 대부분 SQL 준수 -> 표준에 맞춰 관리 및 운용 가능

# 스키마와 인스턴스

&ensp;스키마(Schema)<br/>
* 데이터베이스에 저장되는 데이터 구조와 제약조건을 정의한 것
    - 데이터베이스 내의 속성들의 집합
    - 릴레이션의 테이블명을 포함한 속성들의 집합

<p align="center"><img src="/assets/img/database/1주차/1-6.png" width="600"></p>

&ensp;인스턴스(Instance)<br/>
* 스키마에 따라 데이터베이스에 실제로 저장된 값
* 튜플(Tuple)들의 집합

<p align="center"><img src="/assets/img/database/1주차/1-7.png" width="600"></p>

# 데이터베이스 3단계 구조

&ensp;하나의 데이터베이스에 대해 서로 다른 관점을 가지고 있는 여러 사용자들이 효율적으로 사용할 수 있도록 3단계로 구성<br/>
&ensp;ANSI-SPARC 구조<br/>
```
외부 단계
개념 단계
내부 단계
```

<p align="center"><img src="/assets/img/database/1주차/1-8.png" width="600"></p>

&ensp;1. 외부 스키마(External Schema)<br/>
* 여러 개 존재 가능 -> 다양한 사용자
* 개별 사용자/응용 프로그램 관점에서 이해하고 표현
* 하나의 데이터베이스에 여러 개의 외부 스키마(External Schema) 존재 가능
    - 관심있는 부분을 설명하고 나머지 세부사항을 모두 숨기는 특성
    - 각 사용자가 생각하는 데이터베이스 모습을 한정된 논리적 구조로 표현
&ensp;사용자 관점<br/>
&ensp;예<br/>
```
학생 → 성적 조회
교수 → 학생 관리
```

&ensp;2. 개념 스키마(Conceptual Schema)<br/>
* 데이터베이스이 전체 구조르르 추상하여 논리적으로 정의하는 단계
* 모든 사용자/응용 프로그램의 전체적인 관점에서 이해하고 표현
    - 테이블 내 모든 속성 포함
* 하나의 개념 스키마(Internal Schema; 물리적 저장 구조의 세부사항을 숨기고 데이터 유형, 속성, 관계, 제약조건 등 정의) 존재

&ensp;조직 전체 데이터 구조<br/>
* 전체 논리 구조
* 데이터 관계 정의

&ensp;예<br/>
```
학생
수강
교수
```

&ensp;3. 내부 스키마(Internal Schema)<br/>
* 데이터베이스 추상화 레벨에서 최하위 단계
* 저장 장치의 관점에서 이해하여 표현
* 하나의 내부 스키마(Internal Schema; 저장 장치에 실제로 데이터베이스가 저장되는 방법 정의) 존재 

&ensp;물리적 저장 구조<br/>
&ensp;정의<br/>
* 레코드 형식
* 저장 방법
* 물리적 순서

&ensp;예<br/>
```
디스크 저장 방식
인덱스 구조
파일 구조
```

# 데이터 모델(Data Model)

&ensp;데이터를 컴퓨터로 표현하기 위해 단순화 및 추상화하여 데이터베이스화<br/>

&ensp;데이터 모델링: 현실 세계에 존재하는 데이터를 컴퓨터 내 데이터베이스로 변환하는 과정<br/>

<p align="center"><img src="/assets/img/database/1주차/1-10.png" width="600"></p>

<p align="center"><img src="/assets/img/database/1주차/1-11.png" width="600"></p>

&ensp;데이터베이스 구축을 위해 두 과정 모두 중요<br/>

&ensp;1. 개념적 데이터 모델<br/>
&ensp;사용자 관점<br/>
* 사용자들이 데이터를 인식하는 방식에 대한 개념 제공
* 개체(Entity), 속성(Attribute), 관계(Relationshiop)로 구성
* 고수준 데이터 모델
* 엔티티-관계 모델(Entity-Relationship Model; E-R Model) 사용

&ensp;대표: `E-R 모델` <br/>
&ensp;구성<br/>
* Entity
* Attribute
* Relationship

&ensp;2. 논리적 데이터 모델<br/>
&ensp;데이터 구조 표현<br/>
* 저수준 데이터 모델과 고수준 데이터 모델의 중간
* 일반 사용자가 생각하는 데이터베이스 모습 또는 구조

&ensp;대표: `관계형 모델`, `계층형 데이터 모델`, `네트워크형 데이터 모델` <br/>

<p align="center"><img src="/assets/img/database/1주차/1-12.png" width="600"></p>
<p align="center"><img src="/assets/img/database/1주차/1-13.png" width="600"></p>
<p align="center"><img src="/assets/img/database/1주차/1-14.png" width="600"></p>

&ensp;3. 물리적 데이터 모델<br/>
&ensp;실제 저장 구조<br/>
* 컴퓨터의 저장 장치에 데이터가 어떠한 형태로 저장되는지에 대한 물리적 구조
* 저수준 데이터 모델
* 데이터베이스가 최적이 성능을 내도록 DBMS 특성에 맞게 구성

# 관계형 데이터 모델(Relational Model)

&ensp;가장 많이 사용하는 데이터 모델<br/>
&ensp;형태: `테이블(2차원 테이블 형태)`<br/>

&ensp;예<br/>

| 학번   | 이름  | 학과 |
| ---- | --- | -- |
| 1001 | 홍길동 | 컴공 |
| 1002 | 김철수 | AI |

&ensp;구성 요소<br/>

<p align="center"><img src="/assets/img/database/1주차/1-15.png" width="600"></p>

&ensp;**릴레이션(Relation)**<br/>
* 하나의 개체에 관한 데이터를 2차원 테이블의 구조로 저장한 것
* 파일 관리 시스템 관점에서 파일에 대응
* 릴레이션 스키마(Realtion Schema)와 릴레이션 인스턴스(Relation Instance)로 구성

&ensp;**속성(Attribute)<br/>
* 릴레이션의 열에 해당
* 파일 관리 시스템 관점에서 필드(Field)에 대응
* 데이터베이스를 구성하는 가장 작은 논리적 단위

&ensp;**튜플(Tuple)**<br/>
* 릴레이션의 행에 해당
* 파일 관리 시스템 관점에서 레코드(Record)에 대응

# 데이터베이스 설계 단계

&ensp;순서<br/>
```
요구사항 분석
↓
개념적 설계
↓
논리적 설계
↓
물리적 설계
↓
구현
```

<p align="center"><img src="/assets/img/database/1주차/1-18.png" width="600"></p>

&ensp;1. 요구사항 분석<br/>
* 사용자의 요구사항을 수집 및 분석하여 개발할 데이터베이스의 용도를 파악하는 단계
* 필요한 데이터를 식벼랗고 처리 방법 등을 고려하여 요구사항 명세서 산출
&ensp;예<br/>

```
회원 정보 관리
상품 관리
주문 관리
```

&ensp;2. 개념적 설계<br/출
* DBMS에 독립적인 개념 스키마를 설계하는 단계
* 요구사항 분석 결과물을 개념적 데이터 모델링을 수행해 개념적 구조로 표현

&ensp;개체와 속성 추출 -> 관계 추출 -> E-R 다이어그램 작성

&ensp;E-R 모델 작성<br/>
* 개념적 데이터 모델링의 결과물
* 기호를 사용하여 현실세계의 데이터 및 관계 표현
* 표기법: 피터 첸, IE(Information Engineering), Barker 

관계형 데이터 모델의 예<br/>
<p align="center"><img src="/assets/img/database/1주차/1-17.png" width="600"></p>

&ensp;3. 논리적 설계<br/>
* 개념적 설계 단계에서 생성한 E-R 다이어그램을 논리적 구조로 표현하여 DBMS에 적합한 릴레이션 스키마(논리적 스키마)로 변환하는 단계
    - E-R 모델과 관계 데이터 모델 표현의 차이가 있어 쉽지 않은 단계

&ensp;규칙<br/>
&ensp;1. 모든 개체는 릴레이션으로 변환한다.<br/>
<p align="center"><img src="/assets/img/database/1주차/1-19.png" width="600"></p>

&ensp;2. 다대다(n:m) 관계는 릴레이션으로 변환한다.<br/>
<p align="center"><img src="/assets/img/database/1주차/1-20.png" width="600"></p>

&ensp;4. 일대일(1:1) 관계는 외래키로 표현한다.<br/>

&ensp;5. 다중 값 속성은 릴레이션으로 변환한다.<br/>
